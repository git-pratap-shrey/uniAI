### **Race-Around Condition in J-K Flip-Flops** The **Race-Around Condition** is a phenomenon observed in
level-triggered J-K flip-flops where the output oscillates between 0 and 1 during a single clock pulse, making the
final state unpredictable. #### **Causes** * **Input Conditions:** It occurs specifically when **J = 1** and **K =
1**. * **Clock Duration:** It is caused by a **long clock pulse duration** (where the clock remains at logic '1' for
an extended period). * **Feedback:** Because the flip-flop is level-triggered, the output is fed back to the input
gates while the clock is still high. This causes the output to toggle repeatedly (0 → 1 → 0...) as long as the clock
remains at '1'. #### **Why it is problematic** In digital circuits, the output must be stable and predictable. During
a race-around condition, the output fluctuates, meaning the state of the flip-flop at the end of the clock pulse is
uncertain. This renders the memory element unreliable. --- ### **Elimination of Race-Around Condition** To ensure the
flip-flop changes state exactly once per clock cycle, the following methods are used: 1. **Edge Triggering:** * By
using **edge-triggered** flip-flops (instead of level-triggered), the circuit only responds to the transition of the
clock signal (rising or falling edge). * Since the transition is instantaneous, the feedback loop does not have time
to cause multiple toggles within the same clock pulse. 2. **Master-Slave Configuration:** * A **Master-Slave J-K
Flip-Flop** consists of two flip-flops: a **Master** and a **Slave**. * **Operation:** * When the clock is high, the
**Master** is active and responds to the inputs, while the **Slave** is disabled (holding its previous value). * When
the clock goes low, the **Master** is disabled, and the **Slave** becomes active, transferring the Master's output to
the final output ($Q$). * **Result:** Because the output only changes when the clock transitions (specifically when
the Slave becomes active), the feedback loop is effectively broken during the active period of the Master, preventing
the oscillation. **Summary for Exam:** * **Condition:** $J=K=1$ in level-triggered circuits. * **Effect:**
Unpredictable output oscillation. * **Solution:** Use **Edge Triggering** or **Master-Slave configuration** to ensure
the output changes only once per clock cycle.

### **SQL Injection (SQLi) Attack** **Definition:** **SQL Injection (SQLi)** is a code injection technique where an
attacker inserts malicious **SQL (Structured Query Language)** code into input fields of a web application. This
manipulates the backend database queries, allowing the attacker to bypass security, access sensitive data, or
modify/destroy database contents. It is one of the most common web hacking techniques. --- ### **How an SQL Injection
Attack Works** The attack typically follows these stages: 1. **Vulnerability Identification:** The attacker identifies
a web application that fails to properly sanitize or validate user input (e.g., login forms, search bars, or URL
parameters). 2. **Crafting the Malicious Query:** The attacker creates a malicious SQL statement. For example, instead
of entering a standard search term, they might enter: `'Harry Potter: DROP Table Books;'`. 3. **Injection:** The
malicious code is submitted through a web page input field. 4. **Execution:** The application processes the input and
executes the modified query against the **Relational Database Management System (RDBMS)**. Because the input was not
validated, the database treats the attacker's code as a legitimate command, leading to unauthorized actions. --- ###
**Types of SQL Injection** * **Classic SQLi:** Direct injection of malicious code into an input field to manipulate
the query. * **Blind SQLi:** Used when the application is vulnerable but does not return the results of the query
directly. The attacker infers information by observing the application's response time or error messages. *
**Second-Order SQLi:** The malicious code is injected and stored in the database first, then executed later when the
application processes that stored data. --- ### **Consequences of SQLi** * **Data Leakage:** Unauthorized access to
sensitive information (usernames, passwords, personal data). * **Data Manipulation/Corruption:** Unauthorized
modification or deletion of database records (e.g., dropping tables). * **Unauthorized Access:** Bypassing
authentication mechanisms to gain administrative control. * **Application Compromise:** Potential for full system
takeover. --- ### **Prevention Measures** To prevent SQL injection, developers must ensure that user input is never
treated as executable code. Key strategies include: * **Proper Input Validation:** Implement strict validation rules
for all user-supplied data to ensure it matches expected formats (e.g., length, type, and character set). *
**Sanitization:** Cleanse user inputs to remove or escape characters that have special meaning in SQL (like `'`, `--`,
or `;`). * **Use of Parameterized Queries (Prepared Statements):** This is the most effective defense. By using
prepared statements, the database treats user input strictly as data, not as part of the executable SQL command,
rendering the injection ineffective. * **Principle of Least Privilege:** Configure the database account used by the
web application with the minimum permissions necessary. It should not have administrative rights to drop tables or
access unnecessary system files. * **Regular Security Audits:** Continuously scan web applications to identify and
patch vulnerabilities before they can be exploited.
