------------------------------------------------------------ ACTUAL ----------------------------------------------------
### **Race-Around Condition in J-K Flip-Flops** The **Race-Around Condition** is a phenomenon observed in
level-triggered J-K flip-flops where the output becomes unstable and unpredictable. * **Definition:** It occurs when
both inputs are high (**J = K = 1**) and the clock pulse remains high for a duration longer than the propagation delay
of the flip-flop. * **Mechanism:** Because of the feedback path from the output to the input, the output $Q_{n+1}$
toggles repeatedly between 0 and 1 as long as the clock pulse is at logic 1. * **Result:** The output oscillates and
fails to settle at a stable state, making the final state of the flip-flop indeterminate when the clock pulse returns
to 0. --- ### **Elimination of the Race-Around Condition** To ensure a stable and predictable output, the race-around
condition must be avoided. This is achieved through the following methods: #### **1. Master-Slave J-K Flip-Flop** This
is the most common solution. It uses two flip-flops connected in series: * **Master Section:** Active when the clock
is high (logic 1). It captures the input data. * **Slave Section:** Active when the clock is low (logic 0). It
receives the output from the Master and transfers it to the final output ($Q$). * **Why it works:** Because the Master
and Slave are triggered at different clock levels, the output changes only once per clock cycle. By the time the Slave
responds, the Master is already disabled, preventing the feedback loop from causing oscillations. #### **2. Edge
Triggering** * Instead of using **level triggering** (where the circuit responds to the entire duration of the clock
pulse), **edge triggering** ensures the flip-flop only changes state at the specific transition of the clock signal
(either the rising edge or the falling edge). * **Why it works:** Since the triggering event is instantaneous, the
input is sampled only for a brief moment, preventing the feedback path from causing multiple state changes within a
single clock pulse. --- ### **Summary Table for Exam** | Feature | Description | | :--- | :--- | | **Condition** | $J
= 1, K = 1$ | | **Triggering** | Level-triggered (Primary cause) | | **Symptom** | Output oscillates between 0 and 1 |
| **Solution 1** | Use **Master-Slave** configuration | | **Solution 2** | Use **Edge-triggering** |

Query: What is a SQL Injection attack and how can it be prevented?
Expanded: a SQL Injection attack and how can it be prevented? web attack database attack
Detected Subject: CYBER_SECURITY | Unit: 3 | Mode: syllabus

------------------------------------------------------------ EXPECTED --------------------------------------------------
It is a web security vulnerability that allows an attacker to interfere with database queries by inserting malicious
SQL code into input fields. Prevention includes using prepared statements (parameterized queries) and input

------------------------------------------------------------ ACTUAL ----------------------------------------------------
### **SQL Injection (SQLi) Attack** **Definition:** **SQL Injection (SQLi)** is a code injection technique where an
attacker inserts malicious **SQL (Structured Query Language)** code into input fields of a web application. Because
SQL is the language used to manage data in **Relational Database Management Systems (RDBMS)**, this attack allows the
adversary to manipulate the backend database, potentially leading to unauthorized access, data theft, or destruction.
--- ### **How an SQL Injection Attack Works** The attack typically follows these stages: 1. **Vulnerability
Identification:** The attacker targets a web application that lacks proper **input validation** or sanitization.
Common entry points include login forms, search bars, and URL parameters. 2. **Crafting the Malicious Query:** The
attacker creates a malicious SQL statement designed to interfere with the application's intended logic. * *Example:*
Instead of a standard search keyword, an attacker might input: `'Harry Potter'; DROP TABLE Books;` 3. **Injection:**
The malicious code is submitted via a web page input field. 4. **Execution:** The web application processes the input
and sends the modified query to the database. The database executes the injected command, which may result in: *
**Data Leakage:** Accessing sensitive information (usernames, passwords, personal data). * **Data Manipulation:**
Modifying or deleting (dropping) entire tables. * **Unauthorized Access:** Bypassing authentication mechanisms. ---
### **Types of SQL Injection** * **Classic SQLi:** Direct injection of malicious code into an input field where the
results are visible to the attacker. * **Blind SQLi:** The application is vulnerable, but the database does not return
direct results. The attacker must infer information by observing timing delays or error-based responses. *
**Second-Order SQLi:** Malicious code is injected and stored in the database, but it is not executed until a later
time when the application retrieves and processes that stored data. --- ### **Prevention Measures** To prevent SQL
injection, developers must implement robust security practices: * **Proper Input Validation:** Never trust user input.
Implement strict validation rules to ensure that only expected data formats are accepted. * **Sanitization:** Cleanse
all user-supplied data to remove or escape characters that could be interpreted as SQL commands (e.g., `'`, `;`,
`--`). * **Use Parameterized Queries (Prepared Statements):** This is the most effective defense. It forces the
database to treat user input as data only, never as executable code, regardless of the characters it contains. *
**Principle of Least Privilege:** Configure database accounts with the minimum permissions necessary. For example, a
web application's database user should not have permission to `DROP` tables. * **Regular Security Audits:**
Continuously scan web applications for vulnerabilities and ensure that input fields interacting with the database are
secured.
